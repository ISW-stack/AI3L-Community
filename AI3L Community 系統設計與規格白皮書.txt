System Design & Architecture Specification for AI3L Community

1. 專案概述 (Project Overview)

專案名稱 (Project Name): AI3L Community (AI in Language Learning and Literacy)

目標受眾 (Target Audience): 跨境學術交流社群（支援中國、台灣及國際使用者）。

語言介面 (UI Language): 僅支援英文 (English Only)。

部署環境 (Deployment): 香港伺服器，單體式虛擬主機。

硬體規格底線 (Minimum Hardware Requirement): 為支撐完整的資料庫與快取服務，伺服器配置不得低於 8 vCPU / 16GB RAM。

跨境法規與隱私合規 (Compliance & Data Governance):

GDPR 匿名化 (Right to be Forgotten): 支援被遺忘權，帳號刪除時採「去識別化 (Anonymization)」覆寫，而非單純的軟刪除。

資料落地宣告: 註冊與首次登入時，強制簽署隱私權同意書，載明資料存放於香港。

強制禁止搜尋引擎索引 (noindex, nofollow, robots.txt Disallow)。

2. 系統架構與技術選型 (System Architecture & Tech Stack)

核心架構: 前後端完全分離 (Decoupled Full-Stack Architecture)。

Frontend: Vue.js 3 (Composition API) + TypeScript + Vite + Tailwind CSS。

Backend: Python + FastAPI (Controller-Service-Model 三層架構)。

Database: PostgreSQL 15+ (使用 asyncpg 全非同步驅動)。

Cache & State: Redis (AOF + RDB 混合持久化)。

Task Queue: Celery + Redis (非同步任務，如 CSV 匯出)。

File Storage: MinIO (自建 S3 相容物件儲存)。

!! 擴容觸發條件 (Scaling Trigger): 初期與單機共存，但考量大檔案 I/O 會延遲 PostgreSQL 的 fsync 導致資料庫效能下降，若並發流量超過 200 或磁碟 I/O 達警戒值，必須將 MinIO 抽離至獨立的儲存實例。

Observability (輕量化可觀測性): 為避免壓垮單機記憶體，拔除本機監控服務，改用 SaaS 解決方案 (Datadog for Metrics, Sentry Cloud for Error Tracking)。

3.2 學術論壇 (Academic Forum)

前端介面與功能需實作以下細節：

發文編輯器 (Post Creation/Editor UI):

實作 Rich Text Editor，必須支援粗體、斜體、文字大小切換，以及透過 /files/upload/editor 非同步插入圖片。

提供「附件上傳區塊」，顯示已上傳檔案列表與刪除按鈕 (限制單檔 20MB)。

必須提供分類選擇 (Category，單選或最多3個) 與關鍵字輸入框 (Keywords/Tags，支援 Enter 鍵建立，最多 15 個)。

提供「允許留言 (Allow Comments)」的開關 (Toggle)。

貼文列表與搜尋 (Post List & Search UI):

列表需顯示標題、作者 (點擊導向 Profile)、分類標籤、發布時間、留言數統計。

頂部提供複合搜尋列：包含 Keyword 搜尋框、Category 下拉選單、Date Range Picker (日期區間選擇器)，以及「AND/OR」邏輯切換開關。

必須實作分頁 (Pagination) 或無限滾動 (Infinite Scroll)。

貼文詳情與歷史 (Post Detail & History):

貼文右上角需提供「Edit History (編輯紀錄)」按鈕，點擊彈出 Modal 顯示過去 60 天的版本差異。

提供檢舉 (Report) 按鈕，彈出視窗要求填寫檢舉原因。

留言互動區 (Comments Section):

顯示於貼文下方，採「扁平化展開 (Flat Thread)」加上引言 (Quote/Reply) 顯示模式。

留言框支援 @使用者名稱 自動補齊 (Autocomplete)，並支援簡易附件上傳與表情符號回應 (Like, Smile, Cry)。

3.3 特別興趣小組 (SIGs - Special Interest Groups)

SIG 首頁與列表 (SIGs Directory):

顯示全站 SIG 列表卡片 (包含 SIG 名稱、簡介、Admin 資訊、成員數)。

Admin+ 畫面右上方具備「Create SIG」按鈕；Member 則顯示「Apply to create SIG (導向管理員聯絡方式)」。

SIG 內部結構:

UI 佈局與學術論壇相似，但移除 Category 選擇與分類過濾功能。

頂部加入「Forms (表單)」的頁籤切換，專供該 SIG 內的活動報名或問卷使用。

提供「Members (成員列表)」頁籤，顯示該 SIG 的正副管理員與曾發文的會員。

3.4 內建表單系統 (Built-in Forms)

表單建立介面 (Form Builder UI - 限 Admin/Sub-admin):

頂部區塊：上傳活動 Banner、設定 Title、Description、Deadline (日期時間選擇器)、Max Respondents (數字輸入框)。

題目建構區塊：可動態新增/刪除題目。每題可設定「題型 (Text, Single-choice, Multiple-choice, Dropdown, File Upload)」、「題目文字」與「是否必填 (Required toggle)」。若為選擇題，需提供動態新增選項 (Options) 的輸入框。

表單填寫介面 (Form Submission UI):

根據 Builder 定義的 Schema 動態渲染 HTML 表單元件。

送出前由前端驗證必填項與檔案格式。若表單已達期限或人數上限，前端需將所有 Input 設為 disabled 並顯示 "Closed" 狀態橫幅。

結果匯出 (Export Data):

管理員視角提供「Export CSV」按鈕。點擊後按鈕轉為 Loading 狀態，底層觸發 WebSocket 或 Polling 查詢 Task 狀態，完成後自動觸發瀏覽器下載。

3.5 站內通知系統 (In-App Notifications)

UI 呈現 (Bell Icon & Dropdown):

頂部導航列 (Navbar) 需常駐一個「鈴鐺圖示」，並以紅點 (Badge) 顯示未讀數量。

點擊鈴鐺展開下拉選單 (Dropdown) 顯示最近的通知列表，或點擊「View All」導向完整通知頁面。

通知互動 (Interactions):

每則通知顯示觸發者頭像、訊息內容與時間。

點擊特定通知時：

呼叫 API 標記為已讀 (is_read = true)，通知樣式轉暗。

使用 Vue Router 跳轉至對應的 Entity (例如：跳轉至特定貼文並滾動到 @提及 你的那則留言)。

即時更新 (Real-time updates):

透過 WebSocket 接收到 NEW_NOTIFICATION 時，前端動態增加未讀計數，並彈出 Toast 提示。

4. 系統限制與安全規範 (Constraints & Security)

4.1 系統化威脅分析與防護 (Threat Modeling & Defense)

XSS 防護 (Rich Text Security): 前端渲染時強制使用 DOMPurify。後端存檔前強制使用 bleach 進行二次清洗。

CSP (Content Security Policy): Nginx 強制配置嚴格 CSP Headers (default-src 'self'; img-src 'self' data: minio.domain.com; script-src 'self';) 阻斷惡意腳本。

CSRF 免疫宣告: 系統不依賴 Cookie 傳遞狀態，統一由前端將 JWT 帶入 Authorization Header，先天免疫 CSRF 攻擊。

CAPTCHA 防重放 (Anti-Replay): 本地圖形驗證碼驗證成功後立即從 Redis 刪除 (DEL captcha:{id})。

CORS Policy (跨域策略): 考量跨境網路，FastAPI 必須嚴格設置 ALLOW_ORIGINS，僅允許官方前端網域（包含未來備用網域），絕對禁用 *，並開啟 allow_credentials=True 支援後續架構。

Password Policy: 強制 8 碼以上，包含大小寫與數字。後端強制使用 Argon2id 或 bcrypt 進行密碼雜湊，嚴禁明文或弱演算法。

UI/UX 限制: 所有文字輸入框字元上限均為 500 字元 (Rich Text 內文除外)。

Rate Limiting (API 請求限流) 與 UX 處理: 全站需透過 Nginx 或 FastAPI (搭配 Redis) 實作 API Rate Limiting。

單一 IP 全站限制：60 req/min。

貼文與留言行為：5 req/min，防堵惡意洗版腳本。

Overflow UX (限流體驗): 當觸發限流時，後端必須回傳 HTTP 429 Too Many Requests 並附帶 Retry-After Header。前端攔截器需捕捉此錯誤，並顯示友善的 Toast 提示（例如：「操作過於頻繁，請在 N 秒後重試 (You are performing actions too frequently. Please try again in N seconds.)」），嚴禁直接鎖死畫面或跳出系統錯誤。

4.2 上傳檔案深度防護 (Upload Exploit Prevention)

Magic Number 驗證: 強制比對檔案特徵碼，建立白名單 (.pdf, .docx, .png, .jpg)。

!! 隱私合規防毒策略 (VirusTotal GDPR Risk):

嚴禁將原始檔案 (Raw File) 上傳至任何外部第三方服務 (包含 VirusTotal) 以避免學術機密與 PII 外洩。

實作方式: Celery 背景任務僅能在本地計算檔案的 SHA-256 Hash，並使用此 Hash 值查詢 VirusTotal API。若為未知 Hash，則單純依賴本地端嚴格的 MIME/Magic Number 防護。

PDF 消毒 (Sanitization): 後端強制過濾 PDF，清除潛在的 JavaScript 與惡意巨集 (Macros)。

4.3 資料庫併發控制 (Concurrency & Cascade Strategies)

樂觀鎖 (Optimistic Locking): 貼文表 (POSTS) 具備 version 欄位。編輯時若版本不符，拋出 409 Conflict 防範覆寫 (Lost Update)。

表單 Schema 凍結: 表單收到第一筆回覆後，Schema 欄位即被強制鎖定 (Immutable)，僅允許修改標題或期限，嚴禁刪修題目。

SIG 刪除的 Cascade 策略: SIG 被刪除 (is_deleted = True) 時，應用層觸發 Event 隱藏其下的 Posts 與 Forms，但不直接修改子物件的 is_deleted 狀態，以保留未來的還原彈性。

4.4 資源配額與上限策略 (Resource Quotas)

為防範系統資源被惡意消耗（DDoS 變種或爬蟲濫用），強制設定以下資源上限：

單一使用者發文限制 (Max Posts per User): 每人每日上限 50 篇貼文。

單一使用者容量限制 (Max Storage per User): 每個帳號（含所有圖片、附件）總容量上限 1GB。超過需向管理員申請擴充。

單一貼文留言限制 (Max Comments per Post): 上限 200 則（已於 3.2 節定義）。

單一 SIG 表單限制 (Max Forms per SIG): 每個 SIG 同時最多允許 20 份 Active (未過期) 表單。

5. 工程開發架構指南 (Engineering Guidelines)

5.1 後端進階規範 (Backend Deep-Dive)

交易一致性 (Transaction Consistency): 任何跨表寫入 (如發文+寫入通知) 必須包覆在 async with db.begin():，保證 ACID 原子性。

防重複提交 (Idempotency):

POST/PUT API 前端需帶入 Idempotency-Key (UUID)。後端存入 Redis 並設定 TTL 為 300 秒 (5分鐘)，以涵蓋跨國網路高延遲與前端自動重試的空窗期。若期間收到相同 Key 回傳 409 Conflict。

Redis Key Schema 定義:

session:{role}:{user_id} -> Value: JWT Token, TTL: 45m/3h/5h/8h

online_count:guest -> Integer (Lua Script 原子操作)

jwt:blacklist:{jti} -> Value: 1 (Token 撤銷黑名單)

JWT 與 Redis Session 雙軌驗證 (Strict Session Validation):

Middleware 不能僅驗證 JWT 簽章。必須同時驗證 JWT 未過期 且 Redis 中存在對應的 Session Key。若 Redis 發生重啟遺失 Key，該 JWT 亦視為失效 (回傳 401)，保證狀態絕對一致。

FTS (Full-Text Search) 實作細節: 強制使用 DB Trigger 更新 tsvector。權重設定：Title 為 'A'，Content 為 'B'。

5.2 版本控制與資料庫遷移策略 (Versioning & Database Migration Policy)

為確保系統演進不中斷，強制遵守以下版控與遷移策略：

API Versioning: 所有路由必須帶有版本號 (如 /api/v1/...)。

App Versioning: 採用語意化版本 (Semantic Versioning, vX.Y.Z) 標示前端與後端 Release。

Database Migration Policy (資料庫遷移策略):

強制工具: 使用 Alembic 進行資料庫版控，嚴禁手動進入生產庫 ALTER TABLE。

向後相容原則 (Backward Compatibility): 所有的 Schema 變更必須是向後相容的（Zero-Downtime Migration）。

嚴禁直接重命名或刪除欄位: 若需更改欄位名稱，必須經過「三階段部署 (Expand -> Migrate Data -> Contract)」：先新增新欄位，程式碼同時寫入新舊欄位並回填舊資料，待下次部署才移除舊欄位。

{

  "captcha_id": "uuid-string-here",

{ 
  "status": "success", 
  "data": { 
    "token": "jwt_string_here",

{ 
  "display_name": "Visiting Scholar",

  "captcha_code": "A1B2" 
}

7. WebSocket 與通知設計 (WebSocket & Notifications)

7.1 WebSocket 生命週期 (Lifecycle & Reliability)

Endpoint: wss://domain.com/api/v1/ws

Server 主提 Ping (Keep-Alive): 改由 Server 每 30 秒主動發送 Ping Frame，Client 底層自動回應 Pong。若 Server 90 秒未收到 Pong 即斷線。

前端可見性處理 (Page Visibility API): 當觸發 document.hidden 時，前端暫停一般輪詢機制；重新 visible 時優雅檢查連線狀態。

重連機制: 前端實作指數退避 (Exponential Backoff)，初始 1s，最大 30s。

7.2 Message Schema Example (動態通知渲染防護)

9. DevOps 與可觀測性 (DevOps & Observability)

9.1 測試策略 (Testing Strategy)

Unit/Integration Tests: 後端使用 pytest。所有 Service 層邏輯需達 80% 覆蓋率。

E2E Tests: 前端使用 Playwright 或 Cypress 測試核心發文與登入流程。

9.2 CI/CD 與部署策略 (CI/CD & Deployment Strategy)

Linting & Formatting: flake8 / black (Python), ESLint / Prettier (Vue)。

Testing: 阻擋失敗的 Merge Request。

Deployment Strategy (部署策略): 採用 Rolling Updates (滾動更新) 以確保零停機部署 (Zero-Downtime Deployment)。若是重大 Schema 變更，需配合前述之向後相容 Migration Policy 分次發布。

9.3 日誌等級與格式 (Log Level & Format)

結構化日誌 (Structured JSON Logging): 強制使用 Loguru 輸出 JSON 格式日誌，禁止純文字 print。

必要欄位 (Required Fields): 每一筆日誌皆需包含：timestamp, level, trace_id (用於跨元件請求追蹤), user_id (若已登入), message。

Log Levels (日誌等級規範):

INFO: 核心業務事件 (如：使用者登入、發布貼文)。

WARNING: 異常但可容忍事件 (如：觸發 Rate Limit、密碼錯誤、404 Not Found)。

ERROR: 系統崩潰與非預期例外 (如：資料庫連線失敗、500 Internal Server Error)。

9.4 SLA 與 SLO 目標 (SLA & SLO Objectives)

SLA (Service Level Agreement): 系統服務目標可用性達 99.9% Uptime。

SLO (Service Level Objectives):

API 延遲 (API Latency): 95% (P95) 的 HTTP API 請求回應時間應小於 300ms (不含大型檔案上傳/下載)。

資料庫查詢 (DB Query Time): 核心列表查詢 (包含關聯與分頁) 執行時間應小於 50ms。
