## Part 1: Systemic Bugs

### B-01 [Critical] DM cleanup tasks missing DB pool initialization
- **File:** `backend/app/tasks/dm_cleanup.py:11-99`
- **Description:** Both `cleanup_dm_expired_files` and `cleanup_dm_expired_text` Celery tasks call `dm_repo` and `user_repo` functions that internally call `get_pool()`, but neither task initializes the DB pool first. Every other task file (`cleanup.py`, `virustotal.py`, `form_autoclose.py`, `recommendations.py`, etc.) calls `_ensure_pool()` before DB access. Additionally, `dm_cleanup.py` defines its own `_run_async()` using `asyncio.new_event_loop()` instead of importing the shared async runner, which invalidates any asyncpg connections bound to previous loops.
- **Impact:** When Celery Beat triggers these tasks, `get_pool()` raises `RuntimeError("Database pool is not initialized")`. Expired DM files and text will **never** be cleaned up, causing unbounded storage growth and violating the 30-day text retention policy.
- **Fix:** Import `_ensure_pool` from `app.tasks.cleanup` and call it at the start of both `_cleanup_files()` and `_cleanup_text()`. Replace local `_run_async` with the shared one.

### B-02 [Critical] Recommendation SQL references nonexistent `sig_members.status` column
- **File:** `backend/app/tasks/recommendations.py:203,207`
- **Description:** The `_RECOMMENDATION_BATCH_SQL` query filters `sig_members` with `sm1.status = 'ACCEPTED'` and `sm2.status = 'ACCEPTED'`. The `sig_members` table schema (migration `c3f8a1b2d4e5`) has NO `status` column — only `role` (with values `MEMBER`, `ADMIN`, `MODERATOR`). All `sig_repo.py` operations use `role`, never `status`.
- **Impact:** PostgreSQL raises `column sm1.status does not exist` error. The entire friend recommendation engine is broken — no users ever receive friend recommendations.
- **Fix:** Remove the `AND sm1.status = 'ACCEPTED'` and `AND sm2.status = 'ACCEPTED'` conditions. All `sig_members` rows represent active memberships (join/leave is insert/delete).

### B-03 [Critical] `file_upload` missing from `_VALID_QUESTION_TYPES` in form validation
- **File:** `backend/app/services/form.py:19-21`
- **Description:** The `_VALID_QUESTION_TYPES` frozenset does not include `"file_upload"`. The `validate_question_schema()` function rejects any form containing `file_upload` questions. However, `_validate_answers()` (line 518), `_validate_file_sizes()` (line 538), `get_form_stats()` (line 240), and the frontend `Question.type` union all explicitly handle `file_upload`.
- **Impact:** Forms with `file_upload` questions cannot be created or updated — the validator blocks them with "invalid type" error. This breaks a documented feature.
- **Fix:** Add `"file_upload"` to `_VALID_QUESTION_TYPES`.

### B-04 [High] DM messages not sanitized — stored XSS risk via backend
- **File:** `backend/app/services/dm.py:22-172`
- **Description:** `send_message()` and `edit_message()` store `content` directly into DB without calling `sanitize_html()`. All other user-generated content paths (posts, comments, forms, album comments) pass through `sanitize_html()` / `nh3.clean()`. While the frontend MessageThread renders via text interpolation (`{{ }}`), any future consumer (admin panel, exports, email notifications) rendering DM content as HTML would execute injected scripts.
- **Fix:** Call `sanitize_html(content)` before storing in `send_message()` and `edit_message()`.

### B-05 [High] DM store not reset on logout — stale private messages visible
- **File:** `frontend/src/stores/auth.ts:57-62`
- **Description:** `clearSession()` resets `notificationStore` and `toastStore` but never calls `dmStore.resetState()`. After logout, the previous user's conversations and messages remain in Pinia state. If another user logs in, they could briefly see the previous user's DM data.
- **Impact:** Privacy leak — one user's DM data visible to the next logged-in user.
- **Fix:** Add `useDMStore().resetState()` in `clearSession()`.

### B-06 [High] MessageInput does not react to `editContent` prop changes
- **File:** `frontend/src/components/dm/MessageInput.vue:19`
- **Description:** `const content = ref(props.editContent ?? '')` initializes once but has no `watch` on `props.editContent`. When the parent changes `editContent` (user clicks "Edit" on a different message), the textarea keeps showing the old message content.
- **Impact:** Users editing different DM messages see stale content in the input.
- **Fix:** Add `watch(() => props.editContent, (val) => { if (val !== undefined) content.value = val })`.

### B-07 [High] DM service uses sync `upload_file` blocking event loop
- **File:** `backend/app/services/dm.py:91,133,278`
- **Description:** `upload_file()` and `delete_file()` are synchronous MinIO client calls used in async context without `run_in_threadpool`. For a 50MB file, the event loop blocks for seconds. All other file operations (`files.py`, `album.py`) correctly use `async_upload_file`.
- **Fix:** Use `await async_upload_file(...)` and `await async_delete_file(...)`.

### B-08 [High] RegisterView missing special character password validation
- **File:** `frontend/src/views/RegisterView.vue:63-76`
- **Description:** `passwordChecks` computed validates length/upper/lower/digit but NOT special characters. The backend requires `[^A-Za-z0-9]` (confirmed in MEMORY.md: "Password policy now requires special character"). Users pass frontend validation but fail on backend.
- **Fix:** Add `special: /[^A-Za-z0-9]/.test(password.value)` to `passwordChecks` and include it in `passwordValid`.

### B-09 [High] DM edit/recall/mark_read endpoints missing rate limiting
- **File:** `backend/app/api/v1/endpoints/dm.py:148-178`
- **Description:** `edit_message`, `recall_message`, and `mark_read` endpoints have no `check_rate_limit()`. A user could spam these rapidly causing excessive DB writes and WebSocket broadcasts.
- **Fix:** Add rate limiting similar to `RATE_LIMIT_DM_SEND`.

### B-10 [High] `delete_album` inconsistent ADMIN authorization
- **File:** `backend/app/services/album.py:145-149`
- **Description:** `delete_album` only allows creator or `SUPER_ADMIN` (`user_role == "SUPER_ADMIN"` at line 146). Platform `ADMIN` users are excluded. All other album write operations (`update_album`, `upload_cover`, `add_member`, etc.) use `user_role in ("ADMIN", "SUPER_ADMIN")`. An ADMIN calling `DELETE /albums/{id}` passes the endpoint role check but gets 403 from the service.
- **Fix:** Change `is_super_admin = user_role == "SUPER_ADMIN"` to `is_site_admin = user_role in ("ADMIN", "SUPER_ADMIN")`.

### B-11 [Medium] DM recall decrements char count before verifying recall success
- **File:** `backend/app/services/dm.py:265-271`
- **Description:** `increment_char_count(conversation_id, -content_len)` is called BEFORE `dm_repo.recall_message()`. If recall fails (line 270), char count is decremented but message isn't recalled — inconsistent state.
- **Fix:** Move char count update after successful recall, or combine in a single transaction with advisory lock.

### B-12 [Medium] DM storage quota incremented before message insert succeeds
- **File:** `backend/app/services/dm.py:91-94`
- **Description:** Storage counter is incremented (line 94) after `upload_file()` without try/catch. If subsequent DB operations fail, storage counter is incremented but file is orphaned.
- **Fix:** Increment storage only after message insert succeeds within transaction.

### B-13 [Medium] Co-authors `/my-posts` missing pagination parameter validation
- **File:** `backend/app/api/v1/endpoints/co_authors.py:27-29`
- **Description:** `page` and `page_size` have no `Query()` constraints (no `ge=1`, `le=` limits). `page=0` causes negative offset; `page_size=999999` dumps entire table.
- **Fix:** Add `page: int = Query(1, ge=1, le=10000), page_size: int = Query(20, ge=1, le=100)`.

### B-14 [Medium] `form_repo.find_standalone` missing ILIKE wildcard escaping
- **File:** `backend/app/repositories/form_repo.py:14-16`
- **Description:** Search parameter `f"%{q}%"` doesn't escape ILIKE special chars (`%`, `_`). Other repos (`user_repo`, `post_repo`) correctly use `_escape_ilike()`.
- **Impact:** Searching for `_` matches any character, `%` matches any sequence.
- **Fix:** Use `_escape_ilike(q)` and add `ESCAPE '\\'` to ILIKE clauses.

### B-15 [Medium] `user_repo.list_all` uses `SELECT *` exposing password_hash in memory
- **File:** `backend/app/repositories/user_repo.py:197,207`
- **Description:** All other user queries use `_USER_COLUMNS` (excluding password_hash) except `find_by_username` (intentional for auth). `list_all` unnecessarily exposes hashes through the service layer.
- **Fix:** Change `SELECT *` to `SELECT {_USER_COLUMNS}`.

### B-16 [Medium] `approve_member` does not verify member belongs to correct album
- **File:** `backend/app/services/album.py:444`
- **Description:** `update_member_status(conn, member_uuid, "ACCEPTED")` only uses `member_uuid` without checking `album_id`. An album admin of Album A could approve pending members for Album B by knowing their `member_id`.
- **Fix:** Pass `album_uuid` to `update_member_status` and add `AND album_id = $3` to the SQL.

### B-17 [Medium] QADetailView TiptapEditor content validated with `.trim()`
- **File:** `frontend/src/views/qa/QADetailView.vue:119,357`
- **Description:** Submit button disabled check uses `!newAnswer.trim()`, but TiptapEditor outputs HTML. `<p></p>`.trim() is non-empty, so empty editor content passes validation.
- **Fix:** Use `isContentEmpty()` helper (already used in PostCreateView and QACreateView).

### B-18 [Medium] DMView optimistic recall rollback uses stale array index
- **File:** `frontend/src/views/DMView.vue:202-224`
- **Description:** Rollback captures `idx` before the async API call. If a WebSocket event adds/removes messages during the call, `idx` points to wrong message.
- **Fix:** Re-find message by ID on rollback: `const rollbackIdx = dmStore.messages.findIndex(m => m.id === messageId)`.

### B-19 [Medium] useInfiniteScroll observer not created if sentinel null at mount
- **File:** `frontend/src/composables/useInfiniteScroll.ts:16-27`
- **Description:** Observer is created in `onMounted`, but if `sentinelRef.value` is null (inside `v-if`), observer never starts. No `watch` on `sentinelRef` to retry.
- **Fix:** Add watcher on `sentinelRef` to create/reconnect observer when element becomes available.

### B-20 [Medium] `album_repo.update_album` silently ignores `None` values
- **File:** `backend/app/repositories/album_repo.py:89-95`
- **Description:** Loop checks `if value is not None` before adding to SET clause. Passing `description=None` to clear a description is silently ignored. Compare with `user_repo.update_profile()` which correctly includes None values.
- **Fix:** Remove `if value is not None:` guard.

### B-21 [Medium] UserProfileView co-authored posts shows identical data as regular posts
- **File:** `frontend/src/views/UserProfileView.vue:99-115`
- **Description:** `fetchCoAuthoredPosts()` calls `listPosts({ author_id: userId.value })` — same query as `fetchPosts()`. No co-author filtering is applied. The "Co-Authored Posts" tab shows the exact same posts as the "Posts" tab.
- **Fix:** Add a `co_author=true` API parameter or filter client-side by `p.co_authors` array.

### B-22 [Medium] SIG description not sanitized on create/update
- **File:** `backend/app/api/v1/endpoints/sigs.py:57-68,90-114`
- **Description:** `req.description` passed directly to service without `sanitize_html()`. Frontend uses `DOMPurify.sanitize()` as safety net, but backend should also sanitize for defense-in-depth.
- **Fix:** Add `sanitize_html(req.description)`.

### B-23 [Low] Event retry deserializes kwargs with wrong types (UUID→str)
- **File:** `backend/app/tasks/event_retry.py:57,80`
- **Description:** Failed events serialized with `json.dumps(default=str)` — UUIDs become strings. On retry, handlers receive string args where UUID was expected, causing asyncpg type errors. Retried events always fail again.
- **Fix:** Add UUID/datetime deserialization in retry path.

### B-24 [Low] Multiple views hardcode `'en-US'` locale for date formatting
- **File:** `frontend/src/components/dm/MessageThread.vue:154,318`, `frontend/src/views/admin/InviteCodesView.vue:102`
- **Description:** Hardcoded locale instead of using i18n-aware utilities from `src/utils/date.ts`.
- **Fix:** Use `formatDate()`/`formatDateTime()` with current locale.

### B-25 [Low] PostCard directly mutates prop `comment_count`
- **File:** `frontend/src/components/PostCard.vue:173`
- **Description:** `props.post.comment_count++` directly mutates a prop, violating Vue one-way data flow. Works by reference accident but fragile.
- **Fix:** Emit event to parent instead.

### B-26 [Low] Application review audit emit not wrapped in try/except
- **File:** `backend/app/api/v1/endpoints/applications.py:87-96`
- **Description:** Audit `emit()` not wrapped in try/except. If Redis fails, 500 error despite successful review. All other audit emits use try/except.
- **Fix:** Wrap in try/except like other admin endpoints.

### B-27 [Low] DM filename not sanitized — storage key namespace pollution
- **File:** `backend/app/services/dm.py:90`
- **Description:** `f"dm/{sender_id}/{uuid.uuid4()}_{file_name}"` uses raw filename. Characters like `/` could create keys outside `dm/` prefix, interfering with cleanup jobs.
- **Fix:** Use only the extension: `f"dm/{sender_id}/{uuid.uuid4().hex}{ext}"`.

## Part 2: Security Vulnerabilities

### S-01 [Critical] DM content not server-side sanitized — stored XSS
- **File:** `backend/app/services/dm.py:22-172`
- **Type:** OWASP A03:2021 — Injection
- **Description:** Same as B-04. Only user-input-to-DB path lacking `sanitize_html()`.
- **Attack scenario:** `<img src=x onerror=alert(1)>` stored verbatim. Any HTML renderer (admin panel, export, email) executes the script.
- **Fix:** Apply `sanitize_html()` in `send_message()` and `edit_message()`.

### S-02 [High] DM file upload lacks content-type validation and magic byte checking
- **File:** `backend/app/services/dm.py:76-99`
- **Type:** OWASP A04:2021 — Insecure Design
- **Description:** DM file uploads bypass `validate_editor_file()` and `validate_magic_number()`. Any file type (executables, HTML with JS, SVGs) can be uploaded. Editor and avatar uploads both have magic byte validation.
- **Attack scenario:** Upload `.html` file with JavaScript via DM. Presigned URL serves it inline in browser. Script executes in MinIO origin.
- **Fix:** Add file type allowlist + magic byte validation, or set `Content-Disposition: attachment` on presigned URLs.

### S-03 [High] DM attachment filename path traversal in MinIO key
- **File:** `backend/app/services/dm.py:90`
- **Type:** OWASP A01:2021 — Broken Access Control
- **Description:** Raw filename from `UploadFile.filename` used in MinIO key without sanitization. While MinIO uses flat namespaces, unsanitized filenames bypass the `dm/` prefix-based cleanup and could overlap with other storage prefixes.
- **Fix:** Use only the file extension: `ext = "." + file_name.rsplit(".", 1)[-1].lower() if "." in file_name else ""`.

### S-04 [Medium] IP rate limiting falls back to shared `"unknown"` key
- **File:** `backend/app/api/v1/endpoints/auth.py:89,99,138,213`
- **Type:** OWASP A07:2021 — Auth Failures
- **Description:** When `request.client` is None, all requests share rate limit key `rl:login:unknown`. Behind load balancers without X-Real-IP passthrough, brute-force attempts share a single counter.
- **Fix:** Use `X-Forwarded-For` / `X-Real-IP` headers as primary IP source. Fail closed if no IP determinable.

### S-05 [Medium] Content-Length body size check bypassed by chunked transfer encoding
- **File:** `backend/app/main.py:198-210`
- **Type:** OWASP A05:2021 — Security Misconfiguration
- **Description:** Middleware only checks `Content-Length` header. `Transfer-Encoding: chunked` bypasses the check entirely. In dev, FastAPI is directly accessible on port 18000.
- **Fix:** Use ASGI server-level body size limits or read and limit the actual body stream.

### S-06 [Medium] Missing Content-Disposition on file proxy endpoint
- **File:** `backend/app/api/v1/endpoints/files.py:294-300`
- **Type:** OWASP A03:2021 — Injection
- **Description:** File proxy returns content with original content-type but no `Content-Disposition: attachment` header. If content-type is `text/html`, browser renders inline in application origin.
- **Fix:** Add `Content-Disposition: attachment` for non-image types. Add `Content-Security-Policy: sandbox`.

### S-07 [Medium] SVG in DM image extensions enables potential script execution
- **File:** `frontend/src/components/dm/MessageThread.vue:126`
- **Type:** OWASP A03:2021 — Injection
- **Description:** `IMAGE_EXTS` includes `'svg'`. SVGs can contain JavaScript. While `<img>` blocks scripts, "Open in new tab" renders SVG with full JS capability.
- **Fix:** Remove `'svg'` from `IMAGE_EXTS`.

### S-08 [Medium] WebP magic number check too permissive
- **File:** `backend/app/core/file_validation.py:19`
- **Type:** OWASP A04:2021 — Insecure Design
- **Description:** Checks only `RIFF` (4 bytes). Any RIFF format (WAV, AVI) passes when uploaded with `.webp` extension. Full signature is `RIFF????WEBP`.
- **Fix:** Also check `data[8:12] == b"WEBP"`.

### S-09 [Low] DM file upload has no file type restriction
- **File:** `backend/app/api/v1/endpoints/dm.py:125-133`
- **Type:** OWASP A04:2021 — Insecure Design
- **Description:** Users can upload executables (.exe, .bat, .ps1) via DM. No client-side or server-side type restriction.
- **Fix:** Add file type allowlist or block dangerous extensions.

### S-10 [Low] `window.open(data.download_url)` without URL validation
- **File:** `frontend/src/composables/useFormExport.ts:102`
- **Type:** Open Redirect
- **Description:** Opens URL from API response without validating origin. If backend response is tampered, arbitrary URL opened.
- **Fix:** Validate URL origin matches expected MinIO domain.

### S-11 [Low] DM MessageInput missing `accept` attribute for file input
- **File:** `frontend/src/components/dm/MessageInput.vue:135-140`
- **Type:** Missing Client-Side Validation
- **Description:** File input has no `accept` attribute. All other upload inputs in the app restrict file types.
- **Fix:** Add `accept="image/*,.pdf,.docx,.xlsx,.pptx,.txt,.csv,.zip"`.

## Key Observations

1. **DM feature is the primary risk area.** The DM system (added 2026-03-17) accounts for **18 of 38 findings** (47%). It has not yet gone through the rigorous security hardening that other features received in the 2026-03-16 audit. All 4 Critical findings are DM-related or in DM-adjacent code.

2. **3 features are completely broken:**
   - Friend recommendations engine (B-02: nonexistent column in SQL)
   - Form `file_upload` question type (B-03: blocked by validator)
   - DM cleanup Celery tasks (B-01: no DB pool initialization)

3. **Non-DM code is well-hardened.** Post/comment/form/album pipelines all use `sanitize_html()`, have rate limiting, validate file uploads with magic bytes, and follow consistent patterns.

4. **Frontend XSS protection is solid.** All `v-html` usages use `DOMPurify.sanitize()`. DM MessageThread correctly uses text interpolation.

5. **The core risk pattern is inconsistency.** The DM feature follows the same architecture but missed security measures (sanitization, file validation, rate limiting, pool initialization) that other features implement.

## Positive Findings (Well-Secured Areas)

- All SQL uses asyncpg parameterized queries — no SQL injection
- Dynamic SQL column names use hardcoded allowlists
- Password hashing: Argon2id with strong policy
- JWT secrets blocked in production via `sys.exit(1)` — verified safe
- CSRF: JTI-bound HMAC-SHA256 with constant-time comparison
- Session: Redis blacklisting + single-session enforcement
- WebSocket: one-time ticket auth with 30s TTL
- HTML sanitization: nh3 with strict allowlists (except DM)
- PDF sanitization: pikepdf strips JavaScript/macros
- nginx: `server_tokens off`, comprehensive CSP, rate limiting
- Frontend: HttpOnly cookies, DOMPurify on all v-html, CSRF interceptor, origin-validated login redirect
